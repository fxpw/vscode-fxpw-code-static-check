# План: переименование в vscode fxpw code static check и ревизия проекта

## Цели

1. Переименовать проект из `vscode-fxpw-php-static-check` в `vscode fxpw code static check`.
2. Убрать позиционирование расширения как инструмента только для PHP.
3. Провести ревизию текущего кода и выявить проблемы в корректности, чистоте и поддерживаемости.
4. Подготовить кодовую базу к расчёту метрик по каждой функции не только для PHP, но и для других поддерживаемых языков.

## Область работ

- Манифест расширения и публичные метаданные
- Ключи настроек и пользовательские описания
- Внутренние имена классов, файлов и diagnostic-коллекций
- Согласованность `README` и `CHANGELOG`
- Текущая архитектура анализатора и качество кода
- Дальнейший дизайн мульти-язычных метрик сложности

## Ограничения и допущения

- При переименовании нужно сохранить текущую рабочую функциональность.
- Пользовательские настройки можно менять без сохранения обратной совместимости: будут введены новые ключи.
- Не трогать несвязанные локальные изменения, которые уже есть в рабочем дереве.
- Текущую реализацию метрик для PHP считать первой итерацией, которую нужно перепроверить, а не финальной архитектурой.

## План работ

### 1. Базовая ревизия и инвентаризация

- Просмотреть текущую структуру исходников, точки входа расширения, настройки и поток формирования diagnostics.
- Собрать список всех захардкоженных имён, идентификаторов и PHP-специфичных допущений.
- Разделить задачи на внешнее переименование и внутренний рефакторинг.

### 2. Стратегия переименования проекта

- Обновить метаданные пакета: `name`, `displayName`, `description`, ссылки на репозиторий, подписи в логах.
- Ввести новые ключи настроек под новое имя проекта без сохранения legacy-ключей.
- Переименовать внутренние типы и файлы, которые слишком жёстко привязаны к PHP и мешают новому позиционированию.

### 3. Ревизия кода

- Проверить жизненный цикл анализатора, подписки на события, владение diagnostics и фильтрацию файлов.
- Найти проблемы в парсинге, использовании `async`, дублировании логики и нейминге.
- Выделить, что можно безопасно почистить в рамках переименования, а что лучше вынести в отдельный этап.

### 4. Дизайн обобщённого анализатора

- Определить языконезависимый интерфейс анализатора для расчёта метрик по функциям.
- Разделить определение языка файла и логику расчёта метрик.
- Оставить PHP как одну из реализаций и добавить следующие целевые языки: `JavaScript`, `TypeScript`, `Lua`, `C++`.

### 5. План по метрикам сложности

- Перепроверить текущие эвристики для `Cognitive Complexity` и `Cyclomatic Complexity`.
- Решить, остаёмся ли на regex-подходе или переходим на parser-based анализ для каждого языка.
- Показывать diagnostics по функциям с настраиваемыми warning-порогами для `Cognitive Complexity` и `Cyclomatic Complexity`.

### 6. Этапы реализации

- Этап A: переименование метаданных и документации
- Этап B: чистка внутреннего нейминга и рефакторинг структуры анализатора
- Этап C: исправления по результатам ревизии с низким риском регрессий
- Этап D: расширение метрик сложности для `PHP`, `JavaScript`, `TypeScript`, `Lua`, `C++`

## Ожидаемые результаты

- Обновлённый бренд проекта и документация
- Краткий отчёт по ревизии с найденными проблемами и рекомендациями
- Переработанная структура анализатора, готовая к нескольким языкам
- Следующий этап реализации метрик по функциям для выбранных языков

## Открытые решения

- Выбрать стартовые значения warning-порогов для `Cognitive Complexity` и `Cyclomatic Complexity`
